# --description--

在上一节课中，我们创建了一个功能分支、在该分支提交了新文件并推送到了远程仓库。接下来要做什么？我们需要在 GitHub 上创建一个 pull request（合并请求）。

Pull request 本质上是请求把你分支中的更改合并到目标分支的操作。它是向开源项目提交代码更改的常用流程，便于项目维护者对你的更改进行审查：他们可以发表评论、提问、建议修改，并（希望）批准并合并你的更改。

回到 GitHub 的仓库页面。

当你推送新分支后，通常会看到一个提示横幅，说明你已推送到某个分支，并带有一个创建 pull request 的按钮。这个按钮可以简化流程，不过我们这里先不使用它，以便在没有显示横幅的情况下你也能熟练完成手动流程。

点击仓库页顶部的 Pull Requests 选项卡，然后点击 “New Pull Request” 来准备你的合并请求。

这里有两个重要术语：`base` 与 `compare`（或称 `head`）。`base` 是目标分支，我们要把更改合并到 `main`，因此可以保持默认。`compare` 则是要合并的分支——在本例中应选择 `feature`。

选择后，界面会刷新，显示 `feature` 上但不在 `main` 上的提交，并展示所谓的 diff（差异），可视化显示你所做的具体改动。

由于本例只是添加了文件，diff 比较简洁；对于更复杂的更改，GitHub 会突出显示行级别的修改、添加和删除。学会阅读 diff 需要练习，但熟悉后非常有帮助。

页面顶部有一个小链接 “compare across forks”。如果你是从 MasterPuti 仓库 fork 出来的，这个功能可以让你把 PR 指向原始的 MasterPuti 仓库，而不仅仅是你 fork 的 `main` 分支。

在此我们暂不讨论 fork 的情况，注意不要随意向别人的项目提交仅为练习目的的 PR，这会给维护者带来额外负担。

确认无误后，点击 “Create Pull Request”。在创建页面下方可以再次查看提交列表与 diff，建议最后检查一遍。

你需要为 PR 填写标题（GitHub 会根据合并的提交或分支名自动生成建议标题）和描述。很多项目会提供预设的 PR 模板，若存在请遵循并填写相应内容。

作为仓库所有者，你可以在右侧设置审查者（reviewers）、负责人（assignees）、标签（labels）、项目（projects）和里程碑（milestones）。向他人项目贡献时通常无法直接设置这些内容（只能请求审查），但在自己的仓库里可以随意探索这些功能。

填写并创建 PR 后，就可以看到实际结果啦！

向他人项目发起 PR 时，会进入代码审查（code review）流程，维护者可以请求修改、提出问题，或批准并合并。作为仓库拥有者则可跳过等待，检查提交历史与变更后直接合并。

合并后记得把这些更改拉取到本地：回到终端，切换到 `main` 分支（`git checkout main`），然后运行 `git pull`。

查看 `git log` 时，你会看到你的提交以及诸如 `Merge pull request #1` 的合并记录。GitHub 提供三种合并策略：

- 默认的 “Merge”：把 head 分支的所有提交带入，并创建一个新的合并提交来记录合并；
- “Squash and Merge”：把 head 分支的所有提交压缩成一个单一提交，并把该提交直接合并到 base 分支，且不创建额外合并提交；
- “Rebase and Merge”：将 head 分支的提交“重新基准化”到 base 分支最新提交之后，然后合并（不创建合并提交）。Rebase 是个较复杂的话题，会在后续详细讲解。

不同项目会选用不同的合并策略，因此你应该始终在新分支上提交更改，避免直接在默认的 `main` 分支上提交，这样在从原始仓库更新代码时不会引起混乱。

现在你已经成功创建并合并了第一个 pull request！
