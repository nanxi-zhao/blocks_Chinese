# --description--

在之前的课程中，你学习了如何向自己的仓库发起拉取请求。你还学习了如何合并那个拉取请求。但你只能这样做是因为仓库属于你——你对自己的仓库有维护者权限，这意味着你有权限合并拉取请求。

然而，当你为别人的开源项目做贡献时，你通常不会有这些权限。相反，你需要等待维护者完成代码审查过程并接受你的贡献。然后他们会处理合并。

但代码审查过程是什么样的呢？

维护者采取的过程会因人而异，并且可能有项目特定的方法。但一般来说，代码审查通常有三个步骤。

第一步是确认 CI（持续集成）是否通过。你将在未来的课程中了解更多关于这方面的内容，所以现在你可以将其视为检查代码是否符合样式指南并通过测试套件的验证。

如果 CI 失败，维护者通常会要求你更新拉取请求以解决失败问题。这些失败应该包括一些你可以查看的报告来了解哪里出了问题，但维护者也可能为你提供更具体的指导来解决问题。

一旦检查通过，维护者就会查看你的实际代码更改。他们可以分析差异视图来看你修改、添加或删除了什么。他们可能会询问你为什么采取特定的方法，所以在准备对话时要深思熟虑。

他们还可能要求你进行额外的修改。我们稍后会讨论这是什么样子的。

最后一步通常是将你的更改拉取到本地。维护者可以运行带有你修改的应用程序，测试功能，并验证没有东西被破坏或出现意外行为。

如果一切看起来都很好，维护者会批准你的拉取请求——有时会有一个"LGTM"评论，意思是"看起来不错"。在你的拉取请求被批准后，维护者准备合并更改时应该会合并它。一些项目可能需要多个批准——例如，MasterPuti 需要两个。

但如果你的拉取请求有什么不正确的地方怎么办？维护者可以进行审查，其中他们请求更改。这些请求可以以几种形式出现。

有时，维护者可能只是留下一个"评论"来提出他们的请求，可能指出你应该更改或更新的内容，但在如何实现方面有更多的自由。

或者，维护者也可能留下一个正式的"请求更改"评论。通常，这样的评论会阻止拉取请求被合并。你需要进行所需的更改，然后请求更改的同一维护者需要重新审查并在请求更多更改或批准你的工作之间做出选择。

当维护者有具体的更改想法时，他们实际上可以通过 GitHub 的 UI 提出更改。当他们使用这样的直接建议时，GitHub 会给你一个按钮，可以直接将建议提交到你的分支。但是，如果有多个这样的建议，你需要切换到文件视图。

从这个视图中，你可以将所有建议（一个接一个）添加到"批处理"中，并一次性提交它们。当有很多建议时，这是非常有用的，因为单独处理它们会导致许多提交。所有的 CI 检查都必须在每个提交上运行，这可能会减慢接受你的 PR 的过程。

如果请求的更改很重要，你可能希望在本地进行更改，将它们提交到用于 PR 的同一分支，然后推送上来的。GitHub 将自动更新你的拉取请求以反映新的提交。

还有几个其他需要注意的地方。

首先，记住维护者也是人。开源项目很少产生足够的收入让维护者全职工作，这意味着他们必须保持工作。对审查过程要有耐心很重要——不要反复 ping 维护者要求重新审查，或试图在他们的审查过程中催促他们。

做贡献可能会很令人兴奋！但不要急于求成。相反，要花时间。确保在本地测试你的更改——运行项目可能有的任何测试套件，运行开发服务器并手动确认你的更改是否有效等等。并且始终遵循你正在工作的特定项目的贡献指南。

如果你记住所有这些，并付出最大的努力，你的贡献应该会顺利进行，你的代码审查应该会得到坚实的批准！
